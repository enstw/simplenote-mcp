Headline correction

v1 of this report inferred that /mnt/user-data/outputs was durable "across instances and devices." An unconfounded two-conversation test (same project) shows that is wrong. The outputs mount is keyed per-conversation. A file written in one conversation is not visible in another, even within the same project. It is durable across VM teardowns within a single conversation; it is not a cross-conversation channel.

01 Cross-conversation persistence test verified

Two conversations in the same project. Conversation A wrote a probe file to its outputs and recorded a token + its boot_id. Conversation B (this one) checked every persistent mount for that file, reading from disk only — not from chat memory.

The token never appeared on any durable mount in conversation B. The only way its v2 probe reached this session at all was as a manual chat upload (landing read-only in uploads). Different boot_id confirms a genuinely separate VM, ruling out "same box still alive." Conclusion: outputs is per-conversation, not per-project.

Mechanism, straight from /proc/mounts: every filestore mount—outputs, uploads, tool_results, transcripts—carries the same conversation id in its rclone source string, and that id changes with the conversation. The namespace is the conversation.

02 Persistence model verified

  • Durable, per-conversation: /mnt/user-data/outputs — the only mount that both accepts writes and keeps them across a teardown. Scoped to the conversation that wrote them.
  • Cross-conversation channel: the project store — files explicitly added to the project appear at /mnt/project (read-only copy) in every conversation. This is what actually crosses conversations. I cannot write to it; only the user adds files.
  • Ephemeral: the VM itself — root ext4, /home/claude, /tmp, /dev/shm. Reset on teardown. Installed packages and scratch files do not carry over.
Still open (inferred, not yet proven by us): that outputs survives a teardown within the same conversation. We proved conversations are isolated from each other; we have not yet forced a reset inside one conversation and re-read the token. Until we do, within-conversation durability rests on the mount being a remote filestore, not on a direct test.

06 Boundary model verified + host caveat

This is an isolated, disposable VM, not your machine. There is no user filesystem here to protect, so reads aren't gated by per-read permission prompts the way the Claude Code CLI gates them on real hardware. The real boundaries are the mounts and the network: read-only filestore inputs, an allow-listed egress, and a per-conversation namespace on the durable store.

Not verifiable from inside: host-level isolation between conversations/tenants, and the teardown guarantee itself. The per-conversation id in the mount source is consistent with isolated namespaces, but isolation at the host is Anthropic's to certify — their documentation is authoritative.
